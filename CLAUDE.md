# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

SyncFit — a hybrid-athlete training-readiness MVP. A user signs in, uploads their Strong app CSV export, enters a recurring weekly plan, and clicks "Analyze readiness" to get an AI verdict on today's session based on trailing 72h strength load plus optional Apple Health context (HRV / RHR / sleep) piped in from a native iOS companion. Next.js 16 (App Router, TypeScript) + Tailwind v4, Neon Postgres via Drizzle, Better Auth (email+password), Vercel AI SDK + Anthropic, deployed on Vercel at `https://syncfit-chi.vercel.app`. The iOS companion lives at `ios/SyncFit/` (Swift / SwiftUI / XcodeGen — see "iOS companion" below).

The authoritative scope/architecture decisions and deferred work live in `docs/superpowers/specs/` — the MVP spec at `2026-05-16-syncfit-mvp-design.md` and the iOS-companion spec at `2026-05-23-ios-companion-app-design.md`. Matching plans sit in `docs/superpowers/plans/`. Read the relevant spec before changing behavior — many "missing" things (endurance/Strava, AI `modifications[]`, per-user timezone, RPE, iOS background sync, dashboard health card) are deliberately deferred there, not oversights.

## Commands

- `npm run dev` / `npm run build` / `npm run start` — Next dev/prod. `next dev`/`next build` auto-load `.env.local`.
- `npm test` — offline unit tests (Vitest). Pure modules only; the LLM is mocked; no DB/network. Must stay green and offline.
- Single unit test: `npx vitest run tests/strong-parser.test.ts` or filter `npx vitest run -t "computeTrailingLoad"`.
- `npm run test:integration` — live integration tests against the real `DATABASE_URL` (import/plan/readiness server-action paths; LLM still injected/mocked). Excluded from `npm test` via `vitest.config.ts`; collected by `vitest.integration.config.ts`. Self-cleaning: creates and deletes synthetic `itest-*` users; safe to re-run; idempotent.
- `npx tsc --noEmit` — type check (no test runner needed).
- `npm run lint` — ESLint 9 flat config (Next core-web-vitals + TypeScript, Prettier-compat). `npm run format` — Prettier write; `npm run format:check` — Prettier verify (CI).
- DB schema apply/generate: `node --env-file=.env.local ./node_modules/.bin/drizzle-kit push` (apply to live Neon) / `... drizzle-kit generate` (offline SQL). Better Auth tables are regenerated with `npx @better-auth/cli@latest generate --config src/auth/auth.ts --output src/db/auth-schema.ts` then pushed.

**CLI tools do not auto-load `.env.local`** (only `next` does). Any drizzle-kit / integration-test / one-off DB script must be run via `node --env-file=.env.local …` or it will throw "DATABASE_URL env var is required" (intentional guard in `src/db/index.ts`). The same guard is why files imported by unit tests must NOT statically import `@/db` — see "iOS companion" pure/loader split for the established pattern.

### iOS companion (`ios/SyncFit/`)

- `cd ios/SyncFit && xcodegen generate` — regenerate `SyncFit.xcodeproj` from `project.yml` (the source of truth; the `.xcodeproj` is gitignored). Run after any `project.yml` edit or when fresh source files are added.
- `xcodebuild test -project SyncFit.xcodeproj -scheme SyncFit -destination 'platform=iOS Simulator,name=iPhone 17 Pro'` — run XCTest suite (or any installed iPhone simulator name; check `xcrun simctl list devices available`).
- `xcodebuild build -project SyncFit.xcodeproj -scheme SyncFit -destination 'platform=iOS Simulator,name=iPhone 17 Pro' -quiet` — build only.
- Archive + TestFlight upload is GUI-only via Xcode → Product → Archive → Distribute. Requires `ios/SyncFit/Signing.xcconfig` (per-developer, gitignored) with a valid `DEVELOPMENT_TEAM`; the 1024×1024 placeholder app icon at `ios/SyncFit/SyncFit/Assets.xcassets/AppIcon.appiconset/` is regeneratable via `swift ios/SyncFit/tools/generate-placeholder-icon.swift`. Version bumps live in `ios/SyncFit/project.yml` (`MARKETING_VERSION` + `CURRENT_PROJECT_VERSION`).

## Environment

`.env.local` (gitignored, never commit; CLI needs `--env-file` to see it). `.env.example` documents the four required vars: `DATABASE_URL` (Neon **pooled** string incl. `?sslmode=require`), `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`, `ANTHROPIC_API_KEY`. In production `BETTER_AUTH_URL` MUST equal the exact deployed origin (scheme+host, no trailing slash) or Better Auth's origin check returns 403 on signup; Vercel preview origins are covered by the `https://*.vercel.app` entry in `src/auth/trusted-origins.ts`.

## Architecture (the parts that span files)

**Thin server actions over pure libs.** `src/app/actions/*.ts` are intentionally thin `"use server"` wrappers: they resolve the session, scope by `userId`, and delegate to a testable module in `src/lib/`. Business logic lives in `src/lib/` (`strong-parser`, `trailing-load`, `ai-engine`, `readiness`, `import-persist`, `plan-store`, `health-signals`, `health-pairing`, `device-auth`, `health-window`) so it can be unit/integration tested without HTTP/auth mocking. Put new logic in a lib, not in an action or component. **API routes** (`src/app/api/*/route.ts`) exist for the iOS companion (`devices/pair`, `health/sync`) and follow the same shape: route handler validates payload with zod, calls a lib, never embeds business logic. Integration tests invoke route handlers directly (`import { POST } from "@/app/api/.../route"`) — no dev server required.

**Pure compute / DB load split.** A lib that touches `@/db` cannot be imported by a unit test, because `src/db/index.ts` throws at module load when `DATABASE_URL` is unset. So pure compute and its DB loader live in different files: `recent-training.ts` (pure) ↔ `readiness.ts` `loadRecentTraining` (DB), and `health-signals.ts` (pure `computeHealthSignals`) ↔ `readiness.ts` `loadHealthSignals` (DB). Follow this split when adding any new aggregator — keep the pure compute importable by Vitest.

**Two Drizzle clients — use the right one.** `src/db/index.ts` (`db`, neon-http driver) for normal single-statement reads/writes. `src/db/tx.ts` (`txDb`, neon-serverless `Pool`+`ws`) is used **only** by the two workout+sets writers — `src/lib/import-persist.ts` (Strong-CSV import) and `src/lib/manual-log.ts` `logStrengthWorkout` (manual entry) — because each must wrap a workout row and its set rows in one interactive transaction (the `unique(userId, contentHash)` slot is consumed by the workout insert, so a partial write would make every retry look like a duplicate and silently drop the sets), and the neon-http driver cannot do interactive transactions. `logEnduranceActivity` stays on `db` (genuinely single-statement). Don't introduce `txDb` for single-statement work.

**Auth & scoping (two models).** Web uses Better Auth email+password — `src/proxy.ts` (Next 16 `proxy` file convention, formerly `middleware`) is a presence-only cookie gate (UX, not the security boundary) over `/`, `/import/:path*`, `/plan/:path*`, `/settings/:path*`. Every server action independently calls `auth.api.getSession` and filters all queries by `session.user.id`; client-callable RPC-style actions return `{ error }` shapes on no-session, form-submit actions `redirect("/login")`. iOS uses a separate **device-token bearer model** for `/api/devices/*` and `/api/health/*` — `src/lib/device-auth.ts` `resolveDeviceUser(req)` reads `Authorization: Bearer <token>`, sha256-hashes it, looks up `device_token` scoped to a userId, and returns `{ userId, deviceId } | null`. Tokens are minted by `POST /api/devices/pair` redeeming a short-lived pairing code from `device_pairing`; plaintext token is returned exactly once and lives only on iOS Keychain (DB stores `sha256(token)`). App tables are in `src/db/schema.ts`; Better Auth tables are generated into `src/db/auth-schema.ts` and re-exported from `schema.ts`. The three new tables (`health_metric`, `device_token`, `device_pairing`) intentionally do NOT FK back to `user.id` — matches the pre-existing pattern (Better Auth owns user lifecycle; we scope at every query site). Trusted origins are computed in `src/auth/trusted-origins.ts` (localhost wildcard is dev-only by design).

**Shared nav & the `(app)` route group.** Signed-in pages live under `src/app/(app)/` (`page.tsx` + `dashboard/`, `progress/`, `plan/`, `import/`, `log/`, `settings/devices/`). The route group does **not** change URLs (`/`, `/progress`, `/plan`, `/import`, `/settings/devices`). `src/app/(app)/layout.tsx` resolves the session (redirects to `/login` if absent) and renders `src/app/(app)/site-nav.tsx` — the floating "pill" nav (brand, links, user-menu dropdown with "Devices" + sign-out) — above page content. `(auth)` and the root layout are intentionally nav-free; don't move these pages back to `src/app/` or add nav to the root layout. Active-link logic is the pure, unit-tested `src/lib/nav.ts` (`NAV_ITEMS`/`isActivePath`); the `proxy.ts` matcher is unchanged because route groups don't affect paths.

**Structured plan.** The weekly plan is structured, not free text: `planned_session` (one row per user per weekday; `description` was renamed `notes`) + a `planned_exercise` child table (`name`, `targetSets/Reps/Weight`, `orderIndex`, cascade-deleted with its session). `src/lib/plan-store.ts` does **replace-on-save per day** (session upsert, then delete+re-insert that day's exercise rows scoped by `userId`+`plannedSessionId`) — non-transactional by design (single-day blast radius; don't wire `txDb`). `applyProgressionDecision` writes an accepted suggestion's weight/sets/reps back onto the matching `planned_exercise` (fuzzy name match via the shared `src/lib/exercise-match.ts`, also used by the AI prompt).

**Analyze Readiness flow.** proxy → `/` page → `dashboard/today-session` → `actions/analyze.ts` → `src/lib/readiness.ts` `runReadinessAnalysis({userId, now?, generate?})`: resolves today's weekday in `APP_TZ`, loads that day's `planned_session` + `planned_exercise`, then in one `Promise.all` loads `loadRecentTraining`, `getPlanProfile`, and `loadHealthSignals` (the last wrapped in `.catch` returning `undefined` — health context is additive, must never break the AI flow). `analyzeReadiness` receives all four. The AI returns `todayAdjustments[]` (ephemeral, display-only) and `progressionSuggestions[]` (durable; `status` is stamped server-side as `pending`, never by the model, and accepted via `applyProgressionDecision`). The `now`/`generate` params are injected only by tests. `readiness_analysis.loadSnapshot` includes the health-signals payload alongside the trailing-load data so historical analyses remain reproducible. `readiness_analysis` is persisted **only on success**; an AI failure returns `{error}` and writes nothing, and a non-AI (DB) error is genericized so raw driver messages don't leak. `src/lib/ai-engine.ts` dynamically imports `ai`/`@ai-sdk/anthropic` inside `defaultGenerate` so unit tests stay offline; it retries once then throws a friendly `/couldn't analyze/i` error. `buildPrompt` renders a `## Health signals` block when at least one metric is non-missing, and omits the block entirely otherwise (same safe-rollout shape as the optional `goal` line).

**Progress page.** `/progress` renders a per-lift e1RM/volume line chart per tracked lift. Pure compute is `src/lib/progress.ts` (`computeProgress`, `sortSeries`, unit-tested in `progress.test.ts`); the DB load is `loadProgressData` in `src/app/actions/progress.ts` (the action _is_ the loader here — same pure/compute-importable-by-Vitest discipline). Dates are serialized to ISO strings at the client boundary in `page.tsx`.

**Strong CSV import.** `src/lib/strong-parser.ts` is pure: groups rows by `Date` into workouts, splits `(Equipment)` from exercise name, skips cardio rows (no reps + distance/seconds) with warnings, and computes a sha256 `contentHash` per workout. Dedup is enforced by the `unique(userId, contentHash)` constraint + `onConflictDoNothing`; a real DB failure surfaces as a warning, never a false "skipped duplicate".

**AI plan builder.** `src/lib/plan-generator.ts` mirrors `ai-engine.ts`
(dynamic `ai` import, injected `generate`, one retry, friendly
`/couldn't build/i` error). `proposePlanTurnAction` (in `src/app/actions/plan.ts`)
loads current plan + recent training + goal in one `Promise.all` and returns a
per-turn `{reply, proposedPlan, proposedGoal}` (request/response, not streamed).
The plan page is a client shell (`src/app/(app)/plan/plan-workspace.tsx`) owning
`days`+`goal`; the chat (`plan-chat-drawer.tsx`) is ephemeral and only
_proposes_ — nothing persists until the existing `savePlanWeek`, which now also
upserts `plan_profile.goal` (single-statement on `db`, not `txDb`). The durable
free-text goal is threaded into the readiness prompt (`buildPrompt`/
`runReadinessAnalysis`) and is omitted from the prompt when empty (safe rollout).

**iOS companion (Apple Health).** Native Swift / SwiftUI app at `ios/SyncFit/`, generated by **XcodeGen** from `project.yml` (the `.xcodeproj` is gitignored and regenerated on demand). Data flow: web `/settings/devices` mints a 6-char Crockford-style alphanumeric pairing code (`23456789ABCDEFGHJKMNPQRSTUVWXYZ`, ~880M combos; alphabet + length constants live in `src/lib/health-pairing.ts` and are duplicated as private `static let`s in `PairingView.swift`); iOS POSTs the code to `/api/devices/pair` and gets back a 256-bit base64url bearer token (stored in iOS Keychain, sha256-hashed in `device_token`); iOS reads HealthKit (HRV / RHR / sleep) on app launch, runs a per-metric fallback ladder via the pure `MetricPicker` (spec §6), POSTs `{ uploads: [...] }` to `/api/health/sync`. The route does an idempotent batched upsert on `(userId, metricDate, type)` into `health_metric`, with a `[now − 30d, now + 1d]` window check. On the readiness side, `loadHealthSignals` reads the trailing 7-day window, `computeHealthSignals` builds today + 7-day baseline + freshness tags, and the prompt renders the optional `## Health signals` block. **Debug builds** (Run from Xcode) target `http://localhost:3000` with a narrow `NSExceptionDomains[localhost]` ATS exception; **Release builds** (Archive / TestFlight) target the deployed Vercel URL — see `Config.swift`'s `#if DEBUG / #else` split. v1 is foreground-sync only (no `HKObserverQuery` / `BGAppRefreshTask`); dashboard health card and full physiological readiness are deferred to v2 / v3 per spec. The iOS UI is tokenized onto the vendored `ios/SyncFit/SyncFit/Tokens.swift` (canonical v0.5 `DR*` tokens; the old `DesignTokens.swift`/`DS*` names are gone) — reference `DR*` tokens, don't hard-code colors/spacing.

## Non-obvious gotchas

- **Plan editor must stay controlled.** `src/app/(app)/plan/plan-editor.tsx` is a controlled client component on purpose: React 19 calls native `HTMLFormElement.reset()` after any `<form action={serverAction}>` submit, so uncontrolled (`defaultValue`) fields visually revert to their defaults after Save. Controlled inputs (value+onChange from `useState`) are the fix — this now includes the dynamic exercise rows (keyed by a stable `crypto.randomUUID()` id, not the array index). Don't convert plan/auth form fields back to uncontrolled `defaultValue`, and don't rename the field names the bulk `savePlanWeek` action reads: `title-/notes-/modality-{0..6}`, the per-day `rowCount-{dow}`, and exercise rows `ex-{dow}-{row}-{name|sets|reps|weight}` (the row `name` index must stay positional even though the React key is the uuid).
- **Single timezone.** All date math uses `APP_TZ` (`America/New_York`) from `src/lib/units.ts`. No per-user timezone (deliberate).
- **Design system is vendored.** The Dustin Riley design system lives at `src/styles/design/` (`tokens.css` + `components.css`, plus `public/loader-bar-curl.svg` and an unwired `celebrate.js`); `src/app/globals.css` is `@import "tailwindcss";` then `@import "../styles/design/components.css";` plus a thin Tailwind `@theme` bridge that maps utility keys (`rounded-*`, `font-*`, `text-muted-foreground`, `bg-card`, `border-input`, `text-destructive`) onto the design tokens. Reference the tokens (`--bg`, `--surface`, `--rule`, `--primary`, `--link`, `--space-*`, `--radius-*`, `--shadow-*`, fitness/data-viz tokens) and recipes (`.card`/`.card--soft`, `.btn` + `--cta`/`--ghost`/`--secondary`/`--danger`, `.input`, `.seg`, `.chip`, `.readiness`, `.metric`, `.fig` (aligned tabular-nums figures — route weight×reps/data cells through it, don't hand-roll tabular-nums), `.ai-banner`, `.field`/`.field-grid`, `.password-field`, `.site-nav`, `.bubble`, `.sheet`, `.loader`); never hard-code hex/px. Link text uses `--link` (not `--primary`, which fails AA at small sizes). Sentence-case copy, no emoji. Vendored files are kept faithful to the source bundle (Prettier-ignored); the `dustinriley-design` skill and `docs/superpowers/` carry the full design rules.
- **Next 16, not 15** (accepted; recorded in the plan). The `middleware`→`proxy` deprecation is **resolved**: the file is now `src/proxy.ts` exporting `proxy` (Next 16 convention); the build no longer emits the deprecation warning. Don't recreate `src/middleware.ts` or "fix" this again.
- **iOS project is XcodeGen-managed.** `ios/SyncFit/SyncFit.xcodeproj/` and `*.entitlements` are gitignored — regenerated from `project.yml` on every `xcodegen generate`. Edit `project.yml`, not the pbxproj. Adding new Swift files: drop them in `SyncFit/...` and re-run `xcodegen generate` so Xcode picks them up. `Signing.xcconfig` is per-developer and untracked; create it locally with `DEVELOPMENT_TEAM = <your 10-char team id>` before Archive (simulator builds don't need it).
- **HealthKit needs BOTH usage strings, in BOTH plists.** Even though the iOS app only reads HealthKit (`requestAuthorization(toShare: [], read: ...)`), Apple's TestFlight / App Store validator requires both `NSHealthShareUsageDescription` AND `NSHealthUpdateUsageDescription` for any app claiming the HealthKit entitlement. Since commit 9ec2e90 the Info.plist is split per-config — `ios/SyncFit/SyncFit/Info-Debug.plist` (with the `localhost` ATS exception for simulator) and `ios/SyncFit/SyncFit/Info-Release.plist` (HTTPS-only) — and `project.yml` routes `INFOPLIST_FILE` per config. The two files are mirrored by hand (header comments warn to keep non-ATS keys in sync); any new `Info.plist` key must land in both, and removing either HealthKit usage string from either file breaks Distribute validation.
- **Pairing codes are alphanumeric, not numeric.** Use `PAIRING_CODE_ALPHABET` / `PAIRING_CODE_REGEX` from `src/lib/health-pairing.ts` for any code-handling code on the web side; the iOS `PairingView` duplicates the alphabet as a private constant for keystroke filtering. The route accepts trim + lowercase via a `z.preprocess` (clients can submit any case), but the canonical stored form is uppercase.
- **Production URL is `syncfit-chi.vercel.app`** (Vercel auto-suffixed when `syncfit` was taken). iOS Release builds hard-code this in `Config.swift`. Any references to `syncfit.vercel.app` without the `-chi` are stale placeholders.

## Workflow notes

Tests-and-build must be green before a branch is considered done (`npm test` + `npx tsc --noEmit` + `npm run lint` + `npm run format:check` + `npm run build`; `npm run test:integration` when touching server-action/DB paths; `xcodebuild test` from `ios/SyncFit/` when touching iOS code). The user-only `/ship-check` skill bundles the full gate with the conditional integration/iOS branches; prefer the `ios-build-checker` subagent over inline `xcodebuild test` when verifying iOS changes (keeps the main session free of multi-thousand-line build output). Commits are scoped and frequent. The repo uses the superpowers spec→plan→implement workflow; specs/plans in `docs/superpowers/` are the decision record — update them when scope changes rather than silently diverging (post-implementation amendments at the top of a spec file are an established pattern — see the iOS-companion spec's pairing-code amendment for the shape).
